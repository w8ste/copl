***** Pros and Cons of meta interpreters
#+begin_src python
         '''
         If the interpreted and the interpreter language match closely, a meta
         interpreter can be very easy to write

         - lazy vs strict evaluation
         - ...

         A meta interpreter does not help in understtanding the features we are
         implementing
         - Started implementig recursion based on a cyclic environment, which is arguably

         '''

* Substitution
** Definitions
*** Binding Instance:
 #+begin_src python
   '''
   A binding instance of an identifier is the occurence of the identifier that gives it its
   value.
   '''   
 #+end_src

*** Scope:
 #+begin_src python
    '''
    The scope of a binding instance, BI, is the program region, in which instances of the
    identifier refer to the value bound by BI.
    '''   
 #+end_src 

*** Bound instance: 
 #+begin_src python
   '''
   An identifier is bound if it is contained within the scope of a binding instance of its
   name.
   '''   
 #+end_src

*** Free instance:
 #+begin_src python
   '''
   An identifier not contained in the scope of any binding instance of its name is said to be
   free.
   '''   
 #+end_src

*** Substitution:
 #+begin_src python
   '''
   To substitute identifer \'i\' in \'e\' with expression \'v\', replace all free
   instances of \'i\' in \'e\' with \'v\'.
   '''   
 #+end_src 
 - This definition implies the notion of scope, which means, that we only substitute the bindining instance of an identifier within it's scope
 - An inner binding for same identifier name introduces a new scope. The outer binding's scope is shadowed or masked by the inner binding.

*** Alpha-Equivalence
 #+begin_src python
   '''
   Two terms are identical up to consistent renaming of variables.
   - terns have the same structure
   - references resolve to the same declarations
   - but names used for references and declarations may differ
   '''   
 #+end_src
Example: Let(y, bound, body) $\equiv_\alpha$ Let(z, bound, subst(y, Id(z), body) if !freeVars(body).contains(z))

** Two Substitution Regimes
*** Eager substitution (static and dynamic reduction)
#+begin_src python
    '''
    Avoids re-computing the same value several times.
    '''
      {let {x {+ 5 5}} {let {y {- x 3}} {+ y y}}}
    = {let {x 10} {let {y {- x 3}} {+ y y}}}
    = {let {y {- 10 3}} {+ y y}}
    = {let {y 7} {+ y y}}
    = {+ 7 7}
    = 14

    '''
    Notice how \'+ 5 5\' is evaluated directly and only evaluated once. Afterwards x is
    immediatly substituted in \'y's\' let expression.
    '''
 #+end_src

*** Lazy substitution (static reduction)
#+begin_src python
  '''
  The expression may be evaluated multiple times.
  '''
  {let {x {+ 5 5}} {let {y {- x 3}} {+ y y}}}
  = {let {y {- {+ 5 5} 3}} {+ y y}}
  = {+ {- {+ 5 5} 3} {- {+ 5 5} 3}}
  = {+ {- 10 3} {- {+ 5 5} 3}}
  = {+ {- 10 3} {- 10 3}}
  = {+ 7 {- 10 3}}
  = {+ 7 7}
  = 14
  '''
  Notice how \'+ 5 5\' is only evaluated when actually necessary.  
  '''
#+end_src

** Implementing Substitution
#+begin_src scala
    // id -> Identifier, which will be substituted by expression bound in expression body
    def subst(id: String, bound: Exp, body: Exp): Exp = body match {
      case Num(n) => body
      case Bool(b) => body
      case Plus(lhs, rhs) => Plus(subst(id, bound, lhs), subst(id, bound, rhs))
      case Mult(lhs, rhs) => Mult(subst(id, bound, lhs), subst(id, bound, rhs))
      case Div(lhs, rhs) => Div(subst(id, bound, lhs), subst(id, bound, rhs))
      // If y is equal to our identifier, we return our binding bound, else Id(y)
      case Id(y) => if (y == id) bound else Id(y)
      case Let(letId, boundE, bodyE) =>
        if(id == letId) Let(id, subst(id, bound, boundE), bodyE)
        else
          if (!freeVars(bound).contains(letId))
            Let(letId, subst(id, bound, boundE), subst(id, bound, bodyE))
          else
            val z = freshName(freeVars(bound) ++ freeVars(bodyE))
            Let(z, subst(id, bound, boundE), subst(id, bound, subst(letId, Id(z), bodyE)))
    }
#+end_src
This is called capture-avoiding substitution function that implements sound variable replacement. This is fundamental for programming languages and mathematics in general.

** Conclusion
 #+begin_src python
   '''
   Substitution is not practical for implementing programming languages, becuase the resulting
   interpreter requires a full traversel of the body for each variable definition.
   If the program has size n (measured in abstract syntax tree nodes), then the complexity of
   the interpreter is O(n^2)
   '''   
 #+end_src

* Environments
** Environments instead of substitution
#+begin_src python
  '''
  Substitution is inefficient because every variable binding requires traversing and rewriting the entire expression tree. The resulting complexity of substitution for a program with n AST nodes is O(n^2)
  '''
#+end_src

** let and Environments
 #+begin_src python
   '''
   The interpreter receives a store called environment, which maps identifiers to values. We
   represent environments as values of the type Env.

   type Env = Map[String, Int]
   (let (x e) ...) stores a mapping from x to the value of e in environment.

   When the interpreter encounters an id expression, it looks up the corresponding value in
   in the environment. Free variables not in env -> error
   '''   
 #+end_src

** Static vs. Dynamic Scoping

*** Definition Scope (of a name binding)
#+begin_src python
   '''
   The scope of a name binding is the part of the program where the binding is in effect.
   '''   
 #+end_src

*** Definition Static/Lexical Scoping
 #+begin_src python
   '''
   The scope of a name binding is determined syntactically (at compile-time).
   '''   
 #+end_src

*** Definition Dynamic Scoping
 #+begin_src python
   '''
   The scope of a name binding is determined by the execution context (at runtime).
   '''   
 #+end_src

*** Comparison table
| Aspect              | Static Scoping (Lexical Scoping)                                             | Dynamic Scoping                                                    |
|---------------------+------------------------------------------------------------------------------+--------------------------------------------------------------------|
| Scope Determination | Determined by the variable’s position in the source code.                    | Determined by the order of function calls during execution.        |
| Basis of Resolution | Depends on the lexical structure of the program text.                        | Depends on the runtime call stack and execution context.           |
| Time of Resolution  | Usually determined at compile time.                                          | Determined at runtime.                                             |
| Predictability      | Easy to understand since variable bindings can be seen directly in the code. | Harder to reason about because bindings depend on call history.    |
| Dependency          | Independent of how functions are called.                                     | Depends on which function calls the current function.              |
| Modularity          | Supports modular and maintainable programs.                                  | Can lead to fragile programs due to context-dependent behavior.    |
| Usage in Languages  | Used in most modern languages (e.g., C, C++, Java).                          | Rarely used in modern languages; mostly historical or specialized. |

*** interp(,,): Int =
#+begin_src scala
  def interp(expr: F1LAE, funDefs: FunDefs, env: Env): Int =
    expr match {
      case Num(n) => n
      case Add(lhs, rhs) =>
        interp(lhs, funDefs, env) + interp(rhs, funDefs, env)
      case Sub(lhs, rhs) =>
        interp(lhs, funDefs, env) - interp(rhs, funDefs, env)
      case Let(id, lExpr, body) =>
        // NOTE: This computes the value of the identifer,
        //       by interpreting its assignment expression
        val newEnv = env + (id -> interp(lExpr, funDefs, env))
        interp(body, funDefs, newEnv)
      case Id(name) => env(name)
      case App(fun, arg) => funDefs(fun) match {
        case FunDef(param, body) =>
          // NOTE: We do the same as above. The important part is, that env is extended.
          //       This marks the difference between dynamic and static scoping.
          val funEnv = env + (param -> interp(arg, funDefs, env))
          interp(body, funDefs, funEnv)
      }
    }
#+end_src

** Static Scoping

*** Advantages
1. Easier reasoning about programs
2. Better modularity and reliability

*** Disadvantages
1. More complex implementation via closures
2. No implicit parameter passing

#+begin_src scala
  def interp(expr: F1LAE, funDefs: FunDefs, env: Env): Int =
    expr match {
      case Num(n) => n
      case Add(lhs, rhs) =>
        interp(lhs, funDefs, env) + interp(rhs, funDefs, env)
      case Sub(lhs, rhs) =>
        interp(lhs, funDefs, env) - interp(rhs, funDefs, env)
      case Let(id, lExpr, body) =>
        // NOTE: This computes the value of the identifer,
        //       by interpreting its assignment expression
        val newEnv = env + (id -> interp(lExpr, funDefs, env))
        interp(body, funDefs, newEnv)
      case Id(name) => env(name)
      case App(fun, arg) => funDefs(fun) match {
        case FunDef(param, body) =>
          // NOTE: We do the same as above. The important part is, that env is replaced.
          //       This marks the difference between dynamic and static scoping. The
          //       passed environment solely contains the value of the passed argument.
          val funEnv = Map(param -> interp(arg, funDefs, env))
          interp(body, funDefs, funEnv)
      }
    }
#+end_src

* Function Closures
Needed for implementing first-class functions. The key issue they solve is preserving static
(lexical) scoping when functions are treated as values, with all the rights of values.

#+begin_src scala
  interp(Let("x", Num(3),
    App( Fun("y", Add("x" "y")) Num(4))) )
#+end_src

Standard statical scoping would fail here because x is not within the fresh environment. Therefore we need to closures to remember the current environment at function definition time.

** Definition
Consists of the function + environment. Take a lock at the following expression:
#+begin_src scheme
(let (x 3) (fun (y) (+ x y)))
#+end_src

When evaluating the function, the closure would store the environment [x -> 3], the parameter y and the body (+ x y)
